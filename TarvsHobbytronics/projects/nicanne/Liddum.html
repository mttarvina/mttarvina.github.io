Project Liddum: Personal Home Automation System




I. Overview:

Liddum - the only deity who inhabits the realm called Kabunian; communicates directly with humans on earth; mediator between the people and other gods. [Mythological figure: Ifugao]

Project Liddum is a personal Home Automation System deployed in a server machine connected to a household local area network. Uses custom API to emulate IoT communications between server machine and any node device.



II. Custom Node Communications API

Custom APIs were developed to establish communication between Server and Node. In the current approach, the Node is the one constantly initiating the communication and the Server may either respond or just accept data based on the type of API used by the Node. The method on how the Node communicates to the server is by constructing pattern based URLs that contain data or specific command within the URL path. This URL may or may not exist in the server itself as an actual HTML page but the server will be able to detect predefined patterns in the URL that the Node is trying to access. The Server will then respond accordingly.



II.A. Registering new nodes to the network

When a new node is added to the network, it should send an HTTP request to the server and embed the correct register key to the URL together with all relevant device information related to the Node hardware. If the Server determines that the register key is correct, then the Server responds by giving that Node a unique access key to be used only by that Node. The register key is preconfigured by the Admin user.

When registering, the whole URL pattern should look like this:

<server_ip>:<server_port>/<householdName>/node/register/<registerKey>/<nodeType>/<hardwareRev>/<nodeName>

The server response will be in a JSON formatted data that should look like this:

{
    "node": {
        "name": "<nodeName>",
        "accessKey": "123456789abcdefg"
    }
}

NOTES:

  • If the Node device embeds the wrong register key to the URL, then the server will just send an Error 404 response.


II.B. Controlling Node Actuators and Switches

To control the actuators and switches present in the Node hardware, the current approach is for the Node to constantly poll the server regarding the expected status of all of its actuators and switches. An Admin user will usually connect to the server and set the status of these actuator/switches through a dedicated UI. As an example, an Admin user will be able to set the status of "Lights_1" in Node1 to "ON" through a dedicated UI when connecting to the Server. Every X seconds, Node1 will ask the server: "what should be the status of Lights_1 connected to me now?". Once the Admin user confirms the change in status of "Lights_1", the Server will then be able to respond to Node1 that "Lights_1" should be "ON". Otherwise, the Server will just respond that "Lights_1" should still be "OFF" every time Node1 polls for the latest update. When Node1 receives the response from the Server, it will then switch "Lights_1" to "ON".

To be able to initiate a poll request for device status, the Node device will construct a pattern based URL and will send a simple HTTP request to the server.

The whole URL pattern will look like this:

<server_ip>:<server_port>/<householdName>/node/poll/<nodeName>/<accessKey>/status

The server should respond in a JSON formatted data that look like this:

{
    "node":{
        "name": null,
        "access_key": null,
        "active": true,
        "poll_rate": 2,
        "update_rate": 5,
        "socket1": false,
        "socket2": false,
        "socket3": false,
        "socket4": false,
        "stream": false,
        "device_to_stream": null,
        "streaming_port": null
    }
}

NOTES:

  • These actuator parameters will be updated every time the hardware design is updated.
  • If the Server decides to change the Node ID/access key for this specific Node, then the "id"/"access_key" parameter will not be empty.
  • Setting the value of "socketX" to true will turn on the appliance plugged in to that socket in the Node device.
  • The "stream" parameter will signal the Node device to open up a separate socket port defined by "streaming_port" and send data to this port at a much faster rate.
  • The "device_to_stream" will dictate which module attached to the Node will be sending data to the Server.
  • If the Node device embeds the wrong access key to the URL, then the server will just send an Error 404 response


II.C. Getting Sensor Parameters from Node

To update the Server of the real time sensor values measured by the Node, the Node will construct an HTTP request for each sensor peripheral embedded on its hardware. The URL pattern should look like this:

<server_ip>:<server_port>/<householdName>/node/update/<nodeName>/<accessKey>/<paramName>/<paramValue>

Once the server receives this request and verifies the identity of the Node, the Server sends a response to confirm that data was received. The Server response should like this:

{
    "node":{
        "name": "<nodeName>",
        "acknowledged": true
    }
}

The Server response is then read and parsed by the Node. Once it confirms that the previous request was acknowledged by the Server, the Node proceeds to send out another HTTP request for the next sensor parameter.



III. User Access: LAN Connected

This section outlines the coverage and the different ways an Admin User can access the server app through a browser in any device connected to the local area network.



III.A. First Time Setup or Access

When booting up the Server for the first time, the Admin user should access the default setup page to view the default login details. This can be accessed through a default URL:

<server_ip>:<server_port>/liddum/setup

By default, the login info are as follows:

When logged in, the Admin User has access to the Admin tab where admin related settings can be updated. It is recommended to change the household name under the Admin tab. Once change has been submitted, a prompt will flash to indicate successful household name change. The page will then be automatically redirected to an Error404 page within 2 seconds and the Admin User will be logged out of the current session. You should be able to access the new base URL using the updated household name.

If the new household name was changed to "tarvs" for example, then the new base URL of the Server app will now be in:

<server_ip>:<server_port>/tarvs

The Admin User will then be able to login to this page.
